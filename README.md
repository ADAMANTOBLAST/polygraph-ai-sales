# PolygraphAiSales

**Flex-n-roll PRO — сквозная платформа продаж.** Один контур: сайт, Bitrix24, Telegram, телефония и ИИ. Лид не теряется, воронка живёт в CRM, менеджеры и специалисты работают из одной логики — и в переписке, и в голосе.

---

## Кому и зачем

| Кому | Что даёт платформа |
|------|---------------------|
| **Продажи** | Лиды с сайта сразу в Telegram и в Bitrix; понятная очередь и закрепление клиентов; ответы из админки от своего аккаунта. |
| **Руководство** | Полная картина воронки в Bitrix: стадии, ответственные, история касаний, голос и текст в одной цепочке. |
| **Специалисты** (технолог, экономист, диспетчер и др.) | Маршрутизация по ролям из диалога; передача контекста при handoff; CRM всегда знает, кто ведёт сделку. |
| **Маркетинг** | Лендинг с формой, выбор канала связи, акцент на звонок в колл-центр там, где клиенту удобнее телефон. |
| **Эксплуатация** | Один сервис за nginx, деплой и рестарт скриптами, воспроизводимые зависимости. |

## Что внутри платформы (полный контур)

Ниже — не «список из кода», а **как система задумана и продаётся**: всё, что входило в план сквозной автоматизации, описано как **возможности решения**. Техническая детализация — в `docs/`, здесь — цельный продукт.

### Bitrix24 — центр правды

- Лиды и сделки с сайта: создание, конвертация, резервные сценарии, если стандартный путь недоступен.
- Переписка и события в карточке: таймлайн, комментарии, синхронизация диалога из Telegram, догрузка из админки.
- **Воронка под управлением диалога:** автоматический перевод стадий по исходу коммуникации (в т.ч. через служебные маркеры в ответах ИИ).
- **Маршрутизация по специалистам и воронкам:** назначение ответственного по роли и маршруту — из настроек CRM и логики бота, без ручного перетаскивания на каждом шаге.
- Единая точка для руководства: что обещали клиенту, кто ведёт, на какой стадии сделка — в привычном Bitrix.

### Текст, голос, сайт — одна логика

- **Telegram:** пул аккаунтов команды, входящие в личку, ИИ-ассистент с учётом истории, голосовых и изображений, правила передачи между людьми.
- **Телефония:** сценарии Voximplant (в т.ч. голосовой агент, перевод на специалиста), события звонков в журнале и в админке — голос не «отдельный мир», а тот же контур учёта.
- **Сайт:** форма заявки, способы связи, маршрутизация первого касания в Telegram или в CRM в зависимости от выбора клиента.

### Очередь лидов и специалисты

- Распределение новых обращений по очереди между менеджерами, с сохранением справедливости и учётом отпусков и смены линии.
- Повторный клиент идёт к знакомому менеджеру, пока это не меняет бизнес-правило.
- Передача между ролями с понятным клиенту сообщением и сохранением истории для следующего специалиста.

### Единая админка

- Список диалогов, ветки переписки, отправка от имени нужного аккаунта, отключение авто-ИИ там, где нужен только человек.
- Синхронизация состава команды, ролей и текстов для ИИ.
- Журнал голосовых событий, проверка связи с Bitrix, массовая пересылка переписок в CRM при необходимости.

### ИИ как часть процесса продаж

- Ответы с опорой на историю и профиль аккаунта; медиа в работе (голос, картинки) в рамках политики компании.
- Автоматические действия в сделке через встроенные маркеры — без отдельного «робота в Bitrix» для каждого сценария.

### Оперативность и «не проспать клиента»

- Входящие в Telegram у ответственных в реальном времени.
- Фиксация шагов в CRM — основа для контроля SLA, отчётности перед руководством и последующей аналитики по воронке (в т.ч. когда к ней подключите BI или дашборды — данные уже в Bitrix).

### Надёжность внедрения

- Сервис поднимается без простоя фронта из-за долгого коннекта мессенджеров; health-check для мониторинга; аккуратная работа с ошибками CRM при отправке формы.
- Зависимости: верхний уровень в `requirements.txt`, полный закреплённый граф в `requirements.lock.txt` для одинаковых установок на стендах и в проде.

## Как это устроено технически

Клиент приходит с сайта, из Telegram или с телефона. **Бэкенд** (Python, aiohttp) связывает каналы с **Bitrix24** и пулом **Telegram**, **Comet API** даёт ИИ-ответы, **Voximplant** шлёт события на вебхук, **админка** работает через те же API.

Кратко по потоку:

1. Заявка с сайта → лид и сделка в Bitrix, при выборе Telegram — первое касание в мессенджере.
2. Диалог в Telegram → история и дублирование в CRM.
3. Маркеры в ответе ИИ → обновление стадии и ответственного в сделке по правилам.
4. Звонок → сценарий телефонии → запись в журнале и доступ в админке.
5. Менеджер в админке ведёт диалог и при необходимости отключает автоматику на контакт.

Схемы и модули: [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md).

## Быстрый старт

```bash
git clone <repository-url> PolygraphAiSales
cd PolygraphAiSales
python3 -m venv .venv
./.venv/bin/pip install --upgrade pip
./.venv/bin/pip install -r requirements.lock.txt

cp .env.example .env
cp accounts_registry.json.example accounts_registry.json
# Заполните .env (Comet, Bitrix, порт), добавьте sessions/*.session.

./restart.sh
```

`curl -s http://127.0.0.1:8765/health` — готовность Telegram-пула.

Подробнее: [docs/QUICKSTART.md](docs/QUICKSTART.md).
